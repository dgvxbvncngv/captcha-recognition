基于深度学习的验证码识别与安全分析系统

数据模型文档 (Data Model Specification)

1. 概念模型设计 (Conceptual Data Model)

系统的实体关系模型（E-R 模型）主要包含三个核心实体：用户 (User)、验证码识别记录 (History Record) 和 系统审计日志 (System
Log)。

1.1 实体关系陈述

- 用户 (User) 与 识别记录 (History Record)： 属于 1对多 (1:N)
  关系。一个注册用户可以发起多次验证码识别请求，生成多条历史记录；而一条历史记录只能归属于一个特定用户（游客模式下，操作用户 ID 可为空 NULL）。
- 用户 (User) 与 系统审计日志 (System Log)： 属于 1对多 (1:N)
  关系。一个用户（特别是管理员）的多次操作会产生多条对应的系统审计日志。
- 识别记录与系统日志之间无直接关联。

2. 逻辑模型与物理表结构设计 (SQLite/MySQL)

本系统的物理数据存储在本地关系型数据库 SQLite 中。为了支持将来系统向分布式架构（如
MySQL/MariaDB）平滑迁移，本设计在字段类型定义和约束上保持了高度的兼容性。

2.1 用户物理表 (t_user)

- 表说明： 存储系统内注册用户的账户基础数据、密码哈希值以及分配的角色类型。
- 物理设计表：

| 字段名            | 物理类型     | 对应 MySQL 类型 | 约束条件                   | 默认值             | 说明                                            |
| :---------------- | :----------- | :-------------- | :------------------------- | :----------------- | :---------------------------------------------- |
| `id`            | INTEGER      | INT             | PRIMARY KEY, AUTOINCREMENT | \-                 | 用户自增唯一 ID                                 |
| `username`      | VARCHAR(50)  | VARCHAR(50)     | UNIQUE, NOT NULL           | \-                 | 登录用户名                                      |
| `password_hash` | VARCHAR(256) | VARCHAR(256)    | NOT NULL                   | \-                 | 密码哈希串（哈希加盐）                          |
| `role`          | VARCHAR(20)  | VARCHAR(20)     | NOT NULL                   | 'user'             | 角色划分：'admin'（管理员）/ 'user'（普通用户） |
| `is_active`     | INTEGER      | TINYINT(1)      | NOT NULL                   | 1                  | 账号激活状态：1 启用，0 禁用                    |
| `created_at`    | DATETIME     | DATETIME        | NOT NULL                   | CURRENT\_TIMESTAMP | 账号注册创建的时间戳                            |

2.2 验证码识别记录表 (t_history_record)

- 表说明： 记录每一次验证码上传和模型预测推理的关键参数。此表读写频繁，设有专门的复合索引以优化报表查询。
- 物理设计表：

| 字段名             | 物理类型     | 对应 MySQL 类型 | 约束条件                   | 默认值             | 说明                               |
| :----------------- | :----------- | :-------------- | :------------------------- | :----------------- | :--------------------------------- |
| `id`             | INTEGER      | INT             | PRIMARY KEY, AUTOINCREMENT | \-                 | 记录自增唯一 ID                    |
| `user_id`        | INTEGER      | INT             | FOREIGN KEY, NULLABLE      | NULL               | 操作关联的用户 ID，为外键          |
| `image_path`     | VARCHAR(255) | VARCHAR(255)    | NOT NULL                   | \-                 | 原始图片在服务器的相对物理路径     |
| `predicted_text` | VARCHAR(20)  | VARCHAR(20)     | NOT NULL                   | \-                 | 神经网络模型预测的最终字符文本     |
| `confidence`     | REAL         | FLOAT(5,2)      | NOT NULL                   | \-                 | 推理置信度评分（0.00\~ 100.00）    |
| `tesseract_text` | VARCHAR(20)  | VARCHAR(20)     | NULLABLE                   | NULL               | Tesseract 对照组预测的字符文本     |
| `elapsed_time`   | INTEGER      | INT             | NOT NULL                   | \-                 | 预处理与模型推理端到端耗时（毫秒） |
| `created_at`     | DATETIME     | DATETIME        | NOT NULL                   | CURRENT\_TIMESTAMP | 预测记录生成的时间戳               |

- 外键约束说明： FOREIGN KEY(user_id) REFERENCES t_user(id) ON DELETE SET
  NULL（当用户被删除时，保留其历史数据，外键字段自动设为 NULL，确保统计数据的完整性）。
- 物理索引设计：
  - 索引名称： idx_record_user_time
  - 索引类型： 复合索引（Composite Index）
  - 关联字段： (user_id, created_at)
  - 设计目的： 极大提高针对特定用户在特定时间范围内的识别趋势、日调用量的 SQL 查询检索效率。

2.3 系统审计日志表 (t_system_log)

- 表说明： 存储系统管理性操作及敏感接口的审计轨迹。
- 物理设计表：

| 字段名         | 物理类型     | 对应 MySQL 类型 | 约束条件                   | 默认值             | 说明                                            |
| :------------- | :----------- | :-------------- | :------------------------- | :----------------- | :---------------------------------------------- |
| `id`         | INTEGER      | INT             | PRIMARY KEY, AUTOINCREMENT | \-                 | 日志自增唯一 ID                                 |
| `user_id`    | INTEGER      | INT             | NULLABLE                   | NULL               | 操作人用户 ID                                   |
| `ip_address` | VARCHAR(45)  | VARCHAR(45)     | NOT NULL                   | \-                 | 客户端 IP 物理地址（支持 IPv6）                 |
| `action`     | VARCHAR(100) | VARCHAR(100)    | NOT NULL                   | \-                 | 操作事件（如 "Login Attempt", "Delete Record"） |
| `status`     | VARCHAR(20)  | VARCHAR(20)     | NOT NULL                   | \-                 | 状态："SUCCESS" 或 "FAILED"                     |
| `created_at` | DATETIME     | DATETIME        | NOT NULL                   | CURRENT\_TIMESTAMP | 操作发生的时间戳                                |

3. 内存与缓存数据模型设计 (Redis KV Models)

系统引入 Redis 充当内存高速缓存，以处理会话管理和接口频率限制（限流）。

3.1 会话缓存模型 (Session State)

- Key 结构式： captcha:session:<Session_Token>
- 数据结构类型： Hash
- 存活时间 (TTL)： 1800\text{ 秒} (30分钟，滚动滑窗续期)
- 数据模型：
  {
  "user_id": "1002",
  "username": "user_demo",
  "role": "user"
  }

3.2 接口限流计数器模型 (Rate Limit Counter)

- Key 结构式： captcha:rate_limit:<IP_Address>
- 数据结构类型： String (作为原子计数器使用)
- 存活时间 (TTL)： 60\text{ 秒}
- 数据模型： 自增整数值（最大限制值为 60，超出则触发限流逻辑）。

4. 算法内部张量与矩阵模型 (Tensor & Matrix Mapping)

验证码图片在流转和前向推理计算时，会在不同的内存状态和数据维度（Shape）之间进行转换：

+---------------------------------------------------------------------------------+
| 1. 物理图像流 (File Storage)                                                     |
|    - 存储：JPG / PNG 二进制数据。尺寸不定。                                        |
+----------------------------------------+----------------------------------------+
                                         | cv2.imread(..., cv2.IMREAD_GRAYSCALE)
                                         v
+---------------------------------------------------------------------------------+
| 2. NumPy 灰度矩阵 (OpenCV Preprocessing)                                         |
|    - 维度：[H, W]，类型为 uint8 (0-255)。例如：[45, 120]                           |
+----------------------------------------+----------------------------------------+
                                         | cv2.resize(..., (160, 60))
                                         v
+---------------------------------------------------------------------------------+
| 3. 标准尺寸图像矩阵                                                              |
|    - 维度：[60, 160]，类型为 uint8 (0-255)。                                       |
+----------------------------------------+----------------------------------------+
                                         | np.astype(np.float32) / 255.0
                                         v
+---------------------------------------------------------------------------------+
| 4. 归一化浮点矩阵                                                                |
|    - 维度：[60, 160]，数据类型为 float32 (范围：0.0 - 1.0)。                        |
+----------------------------------------+----------------------------------------+
                                         | torch.from_numpy().unsqueeze()
                                         v
+---------------------------------------------------------------------------------+
| 5. PyTorch 模型输入张量 (Model Input Tensor)                                     |
|    - 维度：[1, 1, 60, 160] (即: [BatchSize, Channels, Height, Width])             |
|    - 数据类型：torch.float32                                                     |
+---------------------------------------------------------------------------------+

6. 数据生命周期与清理策略

为防止 SQLite 数据库因历史记录激增而降低检索效率，同时防止图片存储占满物理磁盘，系统制定了以下数据生命周期治理机制：

1. 历史记录保留策略 (Data Retention Policy)：
   - 普通用户： 只保留最近 90 天 内的验证码识别记录和物理原图。
   - 历史冷数据归档： 每天凌晨 3:00，系统定时运行轻量级清理脚本，检索 created_at < date('now', '-90 day')
     的旧数据：
     1. 首先在磁盘上删除对应的物理图像文件。
     2. 然后将 t_history_record 表中的对应旧数据行做删除 DELETE 操作，并运行 VACUUM
        整理文件，保证物理磁盘空间的回收。
2. 系统日志回滚机制 (Log Rotation Policy)：
   - 系统操作审计日志 t_system_log 保留最近 180 天，超过 180 天的数据由脚本自动清除（因不涉及磁盘图片，直接物理 DELETE
     即可）。
