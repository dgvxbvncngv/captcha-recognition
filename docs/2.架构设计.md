基于深度学习的验证码识别与安全分析系统

架构设计说明书 (System Architecture Design Document)

1. 系统总体架构与物理拓扑

本系统采用分层架构设计加组件化模型，将展示层、服务层、推理层与持久层严格隔离。

1.1 物理部署与拓扑结构

在生产环境中，系统采用轻量化、高鲁棒性的物理拓扑进行部署，结构设计如下：

    +------------------------------------+
                    |         客户端浏览器 / 外部 API      |
                    +-----------------+------------------+
                                      | HTTP (Port 80/443)
                                      v
                    +-----------------+------------------+
                    |             Nginx 反向代理          |
                    +--------+------------------+--------+
                             |                  |
       静态资源与上传图片映射  | (Port 80)        | 代理转发 (Unix Socket / Port 8000)
                             v                  v
                    +--------+--------+  +------+--------+
                    |  /data/uploads  |  |    Gunicorn   | (多进程 WSGI 容器)
                    |   (本地磁盘存储) |  |   WSGI Server |
                    +-----------------+  +------+--------+
                                                | Python WSGI Protocol
                                                v
                    +---------------------------+------------------------+
                    |                     Flask 应用程序进程                |
                    |   +--------------------------------------------+   |
                    |   |               1. 路由与控制器               |   |
                    |   +--------------------------------------------+   |
                    |   |               2. 算法推理引擎               |   |
                    |   |      (PyTorch CRNN 单例 / OpenCV / OCR)     |   |
                    |   +--------------------------------------------+   |
                    +---------+----------------------------+-------------+
                              | Redis Protocol             | SQLite Connector
                              v                            v
                    +---------+-------+          +---------+-------+
                    |      Redis      |          |     SQLite      |
                    |  (Session/限流)  |          |   (物理持久化)  |
                    +-----------------+          +-----------------+

- 流量接入层（Nginx）： 负责接收客户端的 HTTP 请求，过滤恶意长连接，开启 gzip 压缩。对于图片静态资源的请求，由 Nginx
  直接读取磁盘目录 /data/uploads 进行静态加速；对于 API 请求，通过 Unix Domain Socket 转发给后端 Gunicorn
  容器。
- 应用服务层（Gunicorn + Flask）： Gunicorn 作为 WSGI 容器运行多进程实例（推荐配置为
  2 \times \text{CPU核心数} + 1）。Flask 框架在容器内部署，不直接面向公网，保证应用服务器的安全。
- 缓存与限流层（Redis）： 单实例 Redis 服务，通过内存哈希、计数器等结构，提供高并发的限流校验和分布式 Session 存储。
- 物理数据层（SQLite）： 本地单文件嵌入式关系型数据库，通过配置 WAL（Write-Ahead
  Logging）模式实现并发读取，负责记录用户与历史流转明细。

3. 核心技术组件设计

3.1 算法推理引擎单例组件 (Inference Engine)

为防止高并发请求下 PyTorch 频繁加载模型参数造成内存崩溃，推理层组件必须设计为线程安全的单例模式。

    +-----------------------------------+
                       |       Client Request Threads      |
                       +-----------------+-----------------+
                                         |
                                         v
                       +-----------------+-----------------+
                       |     InferenceEngine.get_instance()|
                       +-----------------+-----------------+
                                         |
                                  [First Call?]
                                   /
    Yes  /         \ No
                                 v           v
                  +---------------------+   +---------------------+
                  |   Acquire Lock &    |   |  Return Existing    |
                  |   Load Weights      |   |  Engine Instance    |
                  +----------+----------+   +----------+----------+
                             |                         |
                             +------------+------------+
                                          |
                                          v
                       +------------------+----------------+
                       |    image_tensor = preprocess(img)  |
                       +------------------+----------------+
                                          |
                                          v
                       +------------------+----------------+
                       |     with torch.no_grad():         |
                       |         pred = model(image_tensor)|
                       +-----------------------------------+

- 生命周期管理：
  - 在 Flask 应用启动（Application Init）时，主线程自动触发 InferenceEngine 的实例化。
  - 读取位于 /models/crnn_best.pth 的模型权重，并执行 model.eval()。
  - 在内存中保持该实例常驻。
- 线程安全性：
  - 由于 PyTorch 的前向传播（Forward Pass）只读网络权重，不涉及网络内部状态的变化，因此它是天然线程安全的。
  - 使用全局互斥锁（Mutex Lock）保护模型的加载（Initialize）阶段，确保即使多线程同时启动，模型加载也仅执行一次。

3.2 守护网关组件 (Web Gatekeeper)

应用层的安全边界，集成了限流、鉴权、上传校验三个前置过滤器：

1. 限流模块（Rate Limiter）：

   - 基于 Redis 实现。每次 API 请求到达后，以 captcha:limit:`<IP>` 或 captcha:limit:`<UserID>` 为
     Key 在 Redis 中执行原子自增 INCR。
   - 如果是第一个请求，设置过期时间 EXPIRE 60。若在 60 秒内计数器值超过 60，拒绝请求，返回 429 响应。
2. 文件校验模块（File Validator）：

   - 前端发送图片流后，校验器首先读取文件头部前 4 个字节，匹配其十六进制魔术字。
   - PNG 匹配：89 50 4E 47；JPEG 匹配：FF D8 FF。
   - 拦截一切伪造后缀名的可执行脚本，保障后端执行安全。
3. 并发设计与线程安全

后端采用多进程运行环境，需要重点考虑并发冲突：

- 数据读写并发限制（SQLite WAL 模式）：
  - SQLite 默认在多进程写入时会出现锁库现象。系统在连接初始化时，必须显式启用 WAL（Write-Ahead Logging） 模式。
  - 启用命令：PRAGMA journal_mode=WAL;
  - 该模式允许“单写多读”并发，写入数据不直接覆盖数据库文件，而是写入日志文件中，由 SQLite 后台机制合并，极大提升系统的并发写入吞吐量。
- CPU 计算核心配额限制：
  - PyTorch 默认会尽可能占满宿主机的所有 CPU 核心。在高并发 Web 场景下，这会导致单个请求的推理榨干所有
    CPU，从而造成其他网络请求阻塞。
  - 优化策略： 在推理初始化时，强制将单个推理任务的 CPU 线程限制为 1。
  - 限制命令：torch.set_num_threads(1)

5.2 数据库物理表 DDL 定义 (SQLite 语法)

-- 1. 用户表
CREATE TABLE IF NOT EXISTS t_user (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    username VARCHAR(50) UNIQUE NOT NULL,
    password_hash VARCHAR(256) NOT NULL,
    role VARCHAR(20) NOT NULL DEFAULT 'user',
    is_active INTEGER NOT NULL DEFAULT 1,
    created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP
);

-- 2. 验证码识别记录表
CREATE TABLE IF NOT EXISTS t_history_record (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    user_id INTEGER NULL,
    image_path VARCHAR(255) NOT NULL,
    predicted_text VARCHAR(20) NOT NULL,
    confidence REAL NOT NULL,
    tesseract_text VARCHAR(20) DEFAULT NULL,
    elapsed_time INTEGER NOT NULL,
    created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (user_id) REFERENCES t_user (id) ON DELETE SET NULL
);

-- 创建索引以加速历史查询和数据报表生成
CREATE INDEX IF NOT EXISTS idx_record_user_time ON t_history_record(user_id, created_at);

-- 3. 系统日志审计表
CREATE TABLE IF NOT EXISTS t_system_log (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    user_id INTEGER DEFAULT NULL,
    ip_address VARCHAR(45) NOT NULL,
    action VARCHAR(100) NOT NULL,
    status VARCHAR(20) NOT NULL,
    created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP
);

6.2 错误状态码语义规范

| 统一业务码 | HTTP 状态码 | 业务说明                | 异常处理机制                                       |
| :--------- | :---------- | :---------------------- | :------------------------------------------------- |
| `20000`  | 200         | 请求成功                | 客户端渲染核心业务数据。                           |
| `40001`  | 400         | 参数校验失败            | 拒绝向下传递，客户端提示对应字段不合规。           |
| `40101`  | 401         | 认证失效/未登录         | 重定向客户端至`/login` 登录页。                  |
| `40301`  | 403         | 角色权限不匹配          | 阻止越权操作，前端显示“拒绝访问”。               |
| `42901`  | 429         | 请求过载（流控）        | 提示用户“请求频繁，请稍后再试”。                 |
| `50001`  | 500         | 算法推理/底层数据库故障 | 全局捕获异常，阻断代码异常流，避免泄露服务器路径。 |
