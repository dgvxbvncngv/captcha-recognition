基于深度学习的验证码识别与安全分析系统

开发规范文档 (Development Specification Document)

1. 目的与适用范围

为了保证“基于深度学习的验证码识别与安全分析系统”代码的可读性、可测试性、可维护性，以及团队协作的高效性，特制定本开发规范文档。

本规范适用于参与该项目生命周期中所有代码编写、算法研发、接口设计及版本控制的开发人员。

2. Python 语言工程规范 (PEP 8)

项目后端及算法模块统一采用 Python 语言开发，编码规范严格遵循 PEP 8 标准。

2.1 格式化与排版要求

- 代码缩进： 必须使用 4 个空格进行缩进，禁止使用 Tab 键。
- 行宽限制： 单行代码长度原则上不超过 79 个字符。
- 空行使用：

  - 顶级函数（Top-level Functions）和类定义（Class Definitions）之间空 两行。
  - 类内部的方法（Methods）定义之间空 一行。
- 单行多导入禁用： 每次导入仅导入一个模块。

  # 反例

  import os, sys, time

  # 正例

  import os
  import sys
  import time

### 2.2 命名规范

项目遵循严格的匈牙利法与驼峰法结合的命名逻辑，具体如下：

| 标识符类型            | 命名规则                 | 示例                   | 备注说明                         |
| :-------------------- | :----------------------- | :--------------------- | :------------------------------- |
| **模块/文件名** | 全小写，下划线分隔       | `auth_service.py`    | 简短、具名，避免与标准库名冲突   |
| **类名**        | 驼峰命名法（PascalCase） | `InferenceEngine`    | 首字母大写，不包含下划线         |
| **函数与方法**  | 蛇形命名法（snake_case） | `preprocess_image()` | 动词开头，全小写，下划线分隔     |
| **变量/形参**   | 蛇形命名法（snake_case） | `image_tensor`       | 全小写，下划线分隔               |
| **常量**        | 全大写，下划线分隔       | `MAX_IMAGE_SIZE`     | 声明于模块顶部或专门的配置文件中 |

### 2.3 代码注释与 Docstring 规范

* 所有的公共类、核心算法函数、Web 路由接口必须包含 **Google 风格的 Docstring**。
* **函数注释模版：**
  ```python

3. 算法与深度学习开发规范 (PyTorch)

由于深度学习推理代码运行于 Web 进程内部，其内存及线程安全性至关重要，必须遵守以下算法设计规范：

3.1 显式设备绑定与无状态推理

- 设备自适应： 严禁硬编码 cuda 设备。模型必须显式定义设备绑定：
  device = torch.device('cuda' if torch.cuda.is_available() else 'cpu')

* **禁用全局可变状态：** 模型的前向传播推理网络中，严禁修改任何模型层内的全局参数。所有张量计算、切片必须在局部作用域内完成。

### 3.2 严格的模型推理上下文隔离

在进行前向传播推理（Inference）时，必须显式调用评估模式，并将其包裹在梯度禁用上下文中，以阻止 PyTorch 构建动态计算图，极大节约内存开销：

```python
# 正确示例
model.eval()  # 1. 冻结 Dropout 和 BatchNorm 状态
with torch.no_grad():  # 2. 彻底禁用梯度引擎保存，优化内存开销
    output_tensor = model(input_tensor)

3.3 图像通道格式变换安全性

在 OpenCV 与 PyTorch 交互时，由于维度排列方式不同（OpenCV 为 [H, W, C]，PyTorch 为 [C, H, W]），必须使用
transpose 或 permute 转换，严禁直接 reshape 改变空间像素特征分布：

# 反例：直接 reshape 破坏空间拓扑关系
tensor = torch.from_numpy(image_numpy).reshape(1, 3, 60, 160)

# 正例：使用 permute 重构轴顺序
tensor = torch.from_numpy(image_numpy).permute(2, 0, 1).unsqueeze(0)

4. Web 控制层与业务开发规范 (Flask)

4.1 统一异常处理与边界防御

  - Web 路由控制层（Controller）绝不允许向客户端直接暴露底层数据库、算法报错堆栈（Stack Trace）。
  - 必须使用 Flask 的 @app.errorhandler 统一捕获全局运行时异常，并记录至系统错误日志中，向客户端返回业务状态码 50001：
    @app.errorhandler(Exception)
    def handle_global_exception(e):
        # 1. 记录完整堆栈至日志，供运维调阅
        logger.error(f"Global exception caught: {str(e)}", exc_info=True)
        # 2. 组装安全的统一响应体，防止路径和技术细节外泄
        return make_failed_response(code=50001, message="Internal Server Error")


### 4.2 路由声明与请求参数严格校验
*   任何接收客户端输入的路由接口，必须在前置步骤中对请求参数的类型、长度进行边界检查。
*   文件上传必须严格使用白名单过滤扩展名。

## 5. 数据层操作规范 (Database Operations)

### 5.1 严禁 SQL 拼接，防止 SQL 注入风险
对 SQLite 的任何增删改查操作，禁止使用字符串拼接构建 SQL 语句。**必须统一采用参数化查询（Parametric Query）：**

```python
# 反例：存在严重的 SQL 注入安全漏洞
query = f"SELECT * FROM t_user WHERE username = '{user_input}'"
cursor.execute(query)

# 正例：参数化占位符查询
query = "SELECT * FROM t_user WHERE username = ?"
cursor.execute(query, (user_input,))

5.2 Redis Key 命名互斥隔离

为了防止多模块开发下 Redis 键冲突，Key
的命名必须遵循定义好的前缀命名规范：captcha:<module>:<key_name>。禁止动态生成无前缀约束的全局
Key。

6. Git 版本控制与提交流程规范

项目研发统一采用 Git 分支工作流，代码合并至 main 必须通过 Code Review。

6.1 分支命名规范

  - 新功能分支：feature/<issue_id>_desc（如 feature/101_add_crnn_model）
  - 缺陷修复分支：bugfix/<issue_id>_desc（如 bugfix/205_fix_login_timeout）
  - 文档变更分支：docs/desc（如 docs/update_readme）

6.2 约定式提交规范 (Conventional Commits)

Git Commit 提交信息必须按照以下结构编写：

<type>(<scope>): <subject>

  - feat：新增了系统功能（例如：feat(predict): add support for PNG images）。
  - fix：修复了程序 Bug（例如：fix(auth): fix token expiration time mismatch）。
  - docs：仅文档或注释修改（例如：docs(api): update history query API description）。
  - refactor：重构优化代码，既未新增功能也未修复缺陷。

7. 系统日志分级规范 (Logging)

系统日志输出必须统一格式，且需遵循以下日志分级规范：

[时间戳 (YYYY-MM-DD HH:mm:ss)] [日志级别] [进程ID] [模块名] [日志正文]

  - DEBUG 级别： 用于开发调试期间输出详细算法张量形状、预处理阶段数据。生产环境下该级别日志应关闭。
  - INFO 级别： 用于记录系统关键正常流转节点（如 Model Loaded successfully，User ID 1002 logged in）。
  - WARNING 级别： 用于记录不影响系统运行但具有安全、运行隐患的行为（如 Rate limit triggered by
    IP 192.168.1.1）。
  - ERROR 级别： 运行时发生阻断性错误，但不需要服务器宕机（如 Inference engine prediction failed due to
    NaN loss）。
```
