基于深度学习的验证码识别与安全分析系统

测试计划与测试用例说明书 (Test Plan & Test Case Specification)

1. 测试计划与策略概述

为了确保系统的功能完整性、算法高可用性、高并发承载能力以及多层次安全防御能力，特制定本测试计划。

1.1 测试范畴划分

本项目的测试生命周期涵盖以下四个维度：

    +------------------------------------+
                    |             统一测试入口            |
                    +-----------------+------------------+
                                      |
         +----------------------------+----------------------------+
         |                            |                            |
         v                            v                            v
+--------+--------+          +--------+--------+          +--------+--------+
| 1. 业务功能测试  |          | 2. 算法精度测试  |          | 3. 并发压力测试  |
| (登录/上传/历史) |          | (抗噪/基准对比)  |          | (Locust并发压测) |
+-----------------+          +-----------------+          +-----------------+
         |                            |                            |
         +----------------------------+----------------------------+
                                      |
                                      v
                             +--------+--------+
                             | 4. 系统安全测试  |
                             | (沙箱/流控/木马) |
                             +-----------------+

1. 业务功能测试 (Functional Testing)： 验证用户管理、图像上传交互、历史过滤查询、ECharts 态势展示等基础业务流的正确性。
2. 深度学习算法精度测试 (Algorithm Accuracy Testing)：
   评测模型在带干扰、字符扭曲、字符粘连等不同复杂度验证码下的完全匹配准确率（Complete Matching Accuracy）。
3. 系统高并发与负载压力测试 (Performance & Load Testing)： 评估单实例 CPU 推理模式在高并发场景下的 QPS
   极限、耗时漂移曲线以及多线程资源竞争。
4. 系统安全性测试 (Security Testing)： 验证系统对非合规文件（WebShell 木马）上传的阻断能力，以及基于 Redis
   的流控过滤器对高频恶意攻击的防御效能。
5. 系统测试通过与退出判据

项目进入结题和交付阶段，必须满足以下硬性量化指标门槛：

| 指标类型           | 考核项                            | 目标通过标准 (Threshold)                                                                               |
| :----------------- | :-------------------------------- | :----------------------------------------------------------------------------------------------------- |
| **功能缺陷** | 未解决的缺陷（Defects）分布       | 0 个 Blocker（阻断性异常）与 Critical（严重缺陷）；`<br>`允许保留不超过 2 个 Minor（次要 UI 偏差）。 |
| **算法精度** | 复杂级验证码完全匹配率 (Accuracy) | 字符集完全匹配率$\ge 75\%$。                                                                         |

| **代码质量** | 自动化单元测试覆盖率              | 后端 Service 逻辑与 Utils 单元测试覆盖率$\ge 80\%$。                                                 |

3. 详细测试用例设计 (Test Cases)

3.1 业务功能与接口测试 (Functional Testing)

用例编号：TC-FUN-01 | 历史记录组合条件分页查询验证

- 测试目的： 验证系统在多条件组合（时间范围 + 置信度区间）下进行分页检索的历史记录准确度。
- 前提条件：
  1. 用户已登录，且其名下在数据库中已存在 50 条以上的历史识别记录。
  2. 历史记录中包含不同置信度（从 40% 到 95%）以及不同时间戳的数据。
- 输入测试数据：
  - page: 2, limit: 10
  - start_time: "2023-10-01", end_time: "2023-10-24"
  - 置信度筛选区间设定：80.0\% \sim 95.0\%
- 测试步骤：
  1. 打开历史记录主页。
  2. 设置开始日期和结束日期，调整置信度滑动条至 80% 到 95% 区间。
  3. 点击“筛选”按钮，观察页面渲染的数据列表。
  4. 点击分页控件，切换至第 2 页。
- 预期输出结果：
  1. 前端表格仅展现时间戳介于 10-01 至 10-24 之间的记录，且所有记录的置信度值均在 80\% \sim 95\% 范围内。
  2. 分页栏显示总记录数，且第 2 页数据准确无误地异步加载，无页面整页重载现象。
  3. 开发者工具监控网络请求，API /api/v1/history 成功返回 JSON，且业务码 code 为 20000。

3.2 算法精度与抗噪性能测试 (Algorithm Accuracy Testing)

用例编号：TC-ALG-01 | 复杂级高干扰验证码模型完全匹配率测试

- 测试目的： 定量测试模型在多干扰、字符扭曲、部分粘连场景下的字符对齐与识别能力。
- 前提条件： 准备包含 1000 张高干扰且人工标注完毕的验证码图片测试集。
- 测试步骤：
  1. 在测试服务器上运行本地自动化评测脚本 tests/test_model.py。
  2. 脚本循环读取 1000 张测试集图片，将其送入 OpenCV 预处理流水线，随后送入 CRNN 推理单例中。
  3. 捕获每次输出结果，并与 Ground Truth（真实标注标签）进行比对（全字比对，不可存在任何错漏字）。
  4. 统计并输出最终完全匹配准确率数值。
- 预期输出结果：
  1. 模型运行流畅，未发生内存溢出。
  2. 统计输出的整字完全匹配率（Complete Matching Accuracy） \ge 75.0\%。

3.3 系统负载与压力性能测试 (Performance Testing)

用例编号：TC-PER-01 | 基于多用户并发推理时延压力测试

- 测试目的： 测试多用户并发上传图片进行识别时，系统在纯 CPU 单例加载下的最大 QPS 以及耗时漂移情况。
- 前提条件： 后端服务经由 Gunicorn 生产容器启动（配置为 4 个 worker 工作线程），并启动 Redis。
- 输入测试数据：
  - 使用压测脚本模拟并发数：20。
  - 持续请求时长：60 秒（不设间隔循环发送）。
- 测试步骤：
  1. 在测试机上运行 Locust 压测脚本，设置并发用户数为 20，每秒递增（Spawn Rate）设为 5。
  2. 压测脚本不断调用接口 POST /api/v1/predict，并上传随机的 160x60 测试图片。
  3. 监控 Web 服务器 CPU 及内存占用情况，记录压测结果。
- 预期输出结果：
  1. 系统整体吞吐量（QPS）均值 \ge 15 次/秒。
  2. 在 60 秒的持续高载荷下，请求失败率（Failure Rate） \le 1.0\%。
  3. 后端推理进程无死锁，未出现 HTTP 504 Gateway Timeout 异常。

3.4 系统安全性测试 (Security Testing)

用例编号：TC-SEC-01 | 伪造后缀名 WebShell 木马文件上传过滤测试

- 测试目的： 验证系统对假借图片后缀名、内部实为恶意执行脚本的木马文件上传的防范与过滤能力。
- 前提条件：
  1. 系统处于正常开启状态。
  2. 准备测试文件 webshell.png（实为一个写有木马反弹 shell 的 Python/PHP 脚本，只是将文件扩展名强制修改为
     .png）。
- 测试步骤：
  1. 使用 HTTP 测试工具（如 Postman）向接口 /api/v1/predict 以 multipart/form-data
     格式发送图片预测请求。
  2. 将 webshell.png 绑定至 file 字段。
  3. 点击发送请求，观察后端 API 返回的响应状态码及服务器本地 /data/uploads 目录的文件变动。
- 预期输出结果：
  1. 后端文件校验组件（File Validator）的前置魔术字过滤起效，成功识别其物理头部前 4 个字节不符合 PNG 图像规范。
  2. 系统直接阻断上传和保存操作，未在 /data/uploads 下生成任何临时或物理文件。
  3. API 拦截并返回 JSON 错误报文，状态码为 40001，消息提示：“Invalid file format. Only JPG/PNG
     are allowed.”。

用例编号：TC-SEC-02 | API 接口恶意流控触发测试

- 测试目的： 验证 Redis 限流机制能否精准锁定制止恶意高频刷接口行为。
- 前提条件：
  1. Redis 处于正常启动状态。
  2. 限流机制配置参数为：单个 IP 每分钟最大允许预测 60 次。
- 测试步骤：
  1. 运行 Python 限流测试脚本，使用单线程非阻塞循环，在 5 秒内向 API 接口发送 70 次合规图片预测请求。
  2. 监控前 60 次与第 61~70 次请求返回的 HTTP 状态码。
- 预期输出结果：
  1. 前 60 次请求均正常处理，返回状态码 20000。
  2. 自第 61 次请求开始，系统响应头中返回业务状态码 42901。
  3. HTTP 响应状态码变为 429 Too Many Requests，系统直接阻断请求，不调用后端推理资源，保护算力不受高频攻击耗尽。
